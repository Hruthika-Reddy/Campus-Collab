Software Requirements Specification (SRS) is a document that describes what the software will do and how it will perform.

Requirement Characteristics (SMART)

  • Specific - Clear and unambiguous
  • Measurable - Quantifiable where possible
  • Achievable - Realistic with available resources
  • Relevant - Aligned with business objectives
  • Traceable - Can be tracked through development

Verification refers to checking whether the product is being built right, while Validation checks whether the right product is being built.

Coding Best Practices

  • Follow consistent naming conventions
  • Write modular, reusable code
  • Include meaningful comments
  • Keep methods/functions short and focused
  • Handle errors gracefully
  • Write self-documenting code

Black Box Testing examines functionality without knowledge of internal implementation, while White Box Testing uses internal knowledge to design test cases.

Testing Techniques Comparison

Technique Description When to Use
Equivalence Partitioning Divide input data into equivalent classes For input validation testing
Boundary Value Analysis Test boundaries between partitions For range and limit testing
Statement Coverage Execute all statements at least once Unit testing of critical code
Path Coverage Execute all possible paths through code Complex algorithm testing

Software Quality is the degree to which a software product meets its specified requirements and user expectations.

CMM Maturity Levels

  • Level 1 - Initial: Ad hoc processes, unpredictable results
  • Level 2 - Repeatable: Basic project management established
  • Level 3 - Defined: Standardized processes across organization
  • Level 4 - Managed: Quantitative process management
  • Level 5 - Optimizing: Continuous process improvement

CASE Tools Categories

  • Upper CASE: Requirements and design tools
  • Lower CASE: Implementation and testing tools
  • Integrated CASE: End-to-end development support
